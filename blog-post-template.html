Why Your VPC Endpoint Gap Is Costing More Than You Think

Most mid-market AWS environments route significant cloud service traffic — S3, ECR, Secrets Manager — through NAT Gateways by default. Here's what it's actually costing, and why the fix is simpler than you'd expect.

June 12, 2026
8 min read
Synqmatic

If you pull up the NAT Gateway metrics for a typical mid-market AWS environment, you'll find something surprising. A significant portion of the bytes processed — sometimes the majority — aren't going to external APIs or third-party services. They're going to AWS's own services: S3 buckets, ECR registries, Secrets Manager endpoints, and SSM parameter stores.

This traffic doesn't need to touch the NAT Gateway at all. AWS provides VPC endpoints specifically to keep this traffic on the private network. But in most environments we audit, those endpoints either don't exist, or they exist and aren't working correctly.

The result is a consistent, ongoing cost that compounds quietly until someone looks at the bill.

How the problem develops

VPCs get built quickly. The first person who sets one up follows a tutorial, clicks through the wizard, and creates the subnets and route tables they need to deploy something. The NAT Gateway goes in because it's necessary for internet access. The VPC endpoints don't go in because the tutorial didn't mention them.

Six months later, the environment is running production workloads. Every ECS task pulls container images from ECR through the NAT Gateway. Every Lambda function calls Secrets Manager through the NAT Gateway. Every application shipping logs to S3 does so through the NAT Gateway.

The architecture didn't fail. It just never had the endpoints that would make it efficient.

This is the most common finding in our AWS Networking Audits — not because teams don't know about VPC endpoints, but because there's no moment during initial deployment where their absence becomes an obvious problem.

What the gap actually costs

NAT Gateway data processing charges

NAT Gateway charges $0.045 per GB processed in most regions. This charge applies to every byte that passes through the gateway — in both directions. A containerized environment pulling 500 GB of ECR images per day across a fleet of services generates roughly $675/month in NAT processing charges alone, for traffic that the NAT Gateway doesn't need to handle.

The services that matter most

Not all missing endpoints carry the same cost. The high-impact gaps, in rough order of traffic volume for a typical containerized environment:

  • ECR — container image pulls are large and frequent in ECS/EKS environments. Missing the ecr.dkr and ecr.api Interface Endpoints is the single most common high-cost gap.
  • S3 — the S3 Gateway Endpoint is free and eliminates NAT processing for all S3 traffic. Its absence is almost always an oversight rather than a trade-off.
  • Secrets Manager and SSM — every application startup that fetches credentials, every parameter store lookup, every secret rotation call routes through NAT without these endpoints.
  • SQS and SNS — high-throughput messaging architectures generate significant NAT data processing charges when these endpoints are missing.
Note

The S3 and DynamoDB Gateway Endpoints are free — there is no hourly charge, no data processing charge. Their absence is never a cost trade-off. If they're missing, it's an oversight worth fixing immediately.

The DNS complication

Deploying an Interface Endpoint isn't enough on its own. For the endpoint to actually intercept traffic, DNS queries for the service hostname must resolve to the endpoint's private IP addresses rather than the service's public addresses.

The simplest path is enabling privateDnsEnabled = true on each Interface Endpoint. When this is set, AWS automatically creates the Route 53 private hosted zone and the records needed to resolve the service hostname to the endpoint IPs within your VPC.

The complication arises in hub-and-spoke DNS architectures, where a central DNS resolver handles queries across multiple accounts. In these environments, privateDnsEnabled often has to be set to false — which means you need to create and manage the private hosted zones manually. We routinely find environments where the endpoints exist, privateDnsEnabled is false, and no private hosted zones were created. The endpoints are paying their hourly charge while providing no benefit whatsoever.

# Check whether a VPC endpoint has private DNS enabled
aws ec2 describe-vpc-endpoints \
  --filters "Name=vpc-id,Values=vpc-XXXXXXXX" \
  --query 'VpcEndpoints[*].{Service:ServiceName,PrivateDns:PrivateDnsEnabled,State:State}'

If the output shows PrivateDns: false for any Interface Endpoint, verify that the corresponding private hosted zone exists and is associated with your VPC before assuming the endpoint is working.

Verification steps

  1. List all Interface Endpoints in the VPC and check PrivateDnsEnabled for each.
  2. For any endpoint with PrivateDnsEnabled = false, check whether a Route 53 private hosted zone exists for the service's regional endpoint hostname.
  3. From an EC2 instance or ECS task within the VPC, resolve the service hostname and confirm the returned IP is in your VPC's CIDR range, not a public AWS IP.
  4. Cross-reference NAT Gateway CloudWatch metrics after the change — BytesOutToDestination should decrease for the affected service traffic.

The endpoint gap is one of the clearest examples of a problem where the fix is low-effort, the ongoing cost is real, and the issue persists simply because it's invisible until someone looks for it. If you're not sure whether your environment is affected, the NAT Gateway CloudWatch metrics and a quick route table review will tell you within an hour.
