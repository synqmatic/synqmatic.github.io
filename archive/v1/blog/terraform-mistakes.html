The 5 Most Expensive Terraform Mistakes (And How to Fix Them)

📅 December 2024 ⏱ 8 min read ✍️ SYNQMATIC Team ← Back to Blog

We've reviewed a lot of Terraform codebases. Some from startups that built fast and never looked back. Some from enterprises whose infrastructure "just grew" over three years. Some handed over by departed contractors with no docs and a single remote state file shared across all environments.

The same mistakes keep appearing. And they're not syntax errors — they're architectural decisions made early that compound painfully over time. Here are the five we see most, what they cost, and exactly how to fix them.

Before you read

These fixes apply whether you're on AWS, Azure, GCP, or OCI. The patterns are platform-agnostic — the specific resources and provider configurations differ, but the principles are universal.

Mistake 01 / 05

Storing state locally (or in git)

This is the most common issue and the most dangerous. Terraform's terraform.tfstate file is the source of truth for what's been deployed. If it lives on someone's laptop, or worse — committed to a git repository — you're one hard drive failure or bad git push away from catastrophic infrastructure drift.

We've seen teams spend 2–3 days recovering from a corrupted or lost state file. In one case, it meant manually importing 140+ resources into a new state file to bring things back under management.

The fix: Use a remote backend with state locking

HCL — backend.tf (AWS)
terraform { backend "s3" { bucket = "my-company-tf-state" key = "prod/network/terraform.tfstate" region = "eu-west-1" encrypt = true dynamodb_table = "terraform-state-lock" } }

The DynamoDB table handles state locking — preventing two engineers from running terraform apply simultaneously and corrupting the state. Enable versioning on the S3 bucket so you can roll back to previous state versions. On Azure, use Azure Storage with blob leases. On GCS, use a GCS backend with bucket-level locking.

Mistake 02 / 05

Hardcoding secrets and credentials

We still find AWS access keys, database passwords, and API tokens hardcoded in .tf files — sometimes committed to git history where they live forever, even after deletion. A single exposed credential in a public repo has resulted in six-figure cloud bills for companies mining cryptocurrency on their account.

The fix: Use secrets management from day one

HCL — secrets via AWS Secrets Manager
# Never do this # db_password = "MySecretP@ssword123" ← hardcoded # Do this instead data "aws_secretsmanager_secret_version" "db_password" { secret_id = "prod/db/password" } resource "aws_db_instance" "main" { password = jsondecode( data.aws_secretsmanager_secret_version.db_password.secret_string )["password"] }

Also add a .gitignore that excludes *.tfvars files containing sensitive values, and use terraform.tfvars only for non-sensitive variable overrides.

Mistake 03 / 05

No module structure — everything in one giant main.tf

We've inherited codebases with 3,000-line main.tf files. No modules. No separation of concerns. Every resource for every environment, sometimes in the same file. Changes become terrifying because you don't know what else might be affected by a plan.

The fix: Module-first from the start

Directory structure
infrastructure/ ├── modules/ │ ├── networking/ # VPC, subnets, route tables │ ├── compute/ # EC2, ASGs, launch templates │ ├── iam/ # Roles, policies, instance profiles │ └── security/ # Security groups, NACLs, WAF ├── environments/ │ ├── dev/ │ │ ├── main.tf # Calls modules with dev vars │ │ ├── variables.tf │ │ └── backend.tf │ ├── staging/ │ └── prod/ └── README.md

Each environment has its own state file, its own backend config, and references shared modules. Changing the dev environment can't touch prod — they're entirely separate state files.

Mistake 04 / 05

Skipping the plan — applying directly

Running terraform apply without reviewing the plan is the infrastructure equivalent of pushing directly to main without a PR. We've seen production load balancers deleted because someone missed a -replace flag. A 30-second plan review would have caught it.

The fix: Always save and review the plan

GitHub Actions — plan + manual apply
- name: Terraform Plan run: | terraform plan -out=tfplan terraform show -no-color tfplan > plan.txt - name: Post plan to PR uses: actions/github-script@v7 with: script: | const plan = require('fs').readFileSync('plan.txt', 'utf8') github.rest.issues.createComment({ issue_number: context.issue.number, body: `\`\`\`\n${plan}\n\`\`\`` }) # Apply only runs on merge to main, after human review
Mistake 05 / 05

Missing tagging strategy

No tags means no cost attribution, no environment visibility, and no way to automate lifecycle policies. When your cloud bill hits $50K/month and you can't tell which team or product is responsible for which cost, tagging becomes an emergency retrofit project — and retrofitting tags onto hundreds of existing resources while keeping things working is a miserable experience.

The fix: Enforce tags via a default_tags block

HCL — provider default_tags (AWS)
provider "aws" { region = var.region default_tags { tags = { Environment = var.environment # dev / staging / prod Project = var.project_name ManagedBy = "terraform" Owner = var.team_name CostCentre = var.cost_centre } } }

The default_tags block applies these tags to every resource the provider creates — no need to add them to each resource individually. Use AWS Config or Azure Policy to enforce required tags and flag non-compliant resources automatically.

The common thread

All five mistakes share the same root cause: decisions made early for speed that compound into significant risk and cost over time. The fix for each takes between 30 minutes and a day to implement — far less than the time spent recovering from the consequences of skipping them.

If you've inherited an existing codebase with these issues, don't try to fix everything at once. Start with remote state and backend locking (Mistake 1) — it's the highest-risk issue and the fastest to fix. The rest can be addressed incrementally as part of normal sprint work.
